🔬 Lektion 40 ⏱ ~30 Min Lesezeit Evals Engineering

Eval Loop

Prompt ändern → Output besser oder schlechter? Ohne Evals: Raten. Mit Evals: Zahlen. So baust du systematische Qualitätssicherung für AI-Outputs.

1 Warum Evals — der Unterschied zwischen Raten und Wissen

Du änderst den System-Prompt deines Chatbots. Er klingt jetzt "irgendwie besser" — oder doch schlechter? Du testest 5 Beispiele manuell und entscheidest. Das Problem: 5 Beispiele sind keine repräsentative Stichprobe. Du optimierst auf 5, regredierst auf 500.

❌ Ohne Evals
  • Gefühl statt Daten
  • Regressionen unbemerkt
  • Kein Vergleich von Prompt-Versionen
  • Jede Änderung ist ein Glücksspiel
  • Schwer delegierbar
✅ Mit Evals
  • Score vorher: 72% — nachher: 89%
  • Regressionen sofort sichtbar
  • Jede Prompt-Änderung messbar
  • CI/CD für AI-Outputs möglich
  • Automatisierbar, skalierbar
📐 Analogie: Evals sind für AI-Prompts was Unit-Tests für Code sind. Du würdest keinen Code deployen ohne Tests. Warum deployest du Prompts ohne Evals?

Was ist ein Eval-Set?

Ein Eval-Set ist eine Sammlung von Input/Expected-Output-Paaren, gegen die du dein System automatisch testest. Minimum: 20 Beispiele. Ideal: 50-100 für repräsentative Ergebnisse.

// Eval-Case-Struktur
{
  "id": "eval-001",
  "input": "Erkläre mir, wie ihr Onboarding funktioniert",
  "expectedOutput": "Enthält: 3-Schritte, Zeitrahmen, Ansprechpartner",
  "checkFn": "enthält alle 3 Pflicht-Elemente",
  "category": "onboarding",
  "priority": "high"
}

2 Der Eval-Loop — 6 Schritte

1
Eval-Set erstellen
2
Baseline messen
3
Prompt ändern
4
Eval laufen
5
Score vergleichen
6
Commit/Revert

Schritt 1: Eval-Set erstellen — Mindest-20-Regel

20 Beispiele sind das absolute Minimum für statistische Aussagekraft. Sammle Beispiele aus:

  • Real-World-Conversations: Exportiere die ersten 50 echten Nutzer-Anfragen
  • Edge Cases: Fragen die den Bot früher zum Scheitern gebracht haben
  • Happy-Path: Standard-Anfragen die gut funktionieren sollten
  • Adversarial: Versuche den Bot aus dem Rahmen zu bringen

Schritt 2: Baseline messen

Führe Evals gegen den aktuellen Prompt aus und notiere den Score. Das ist deine Baseline — jede zukünftige Änderung wird dagegen gemessen.

Baseline-Messung:
  Prompt v1.0 → Score: 68/100 (68%)
  ✓ 68 Cases passed
  ✗ 32 Cases failed
  Kategorien: onboarding 80%, pricing 55%, support 65%

Schritt 6: Commit oder Revert

✅ Committen wenn:
  • Score ≥ Baseline + 3%
  • Kein Rückschritt in High-Priority-Cases
  • Verbesserung statistisch signifikant
❌ Revertieren wenn:
  • Score ≤ Baseline − 1%
  • Rückschritt bei High-Priority-Cases
  • Verbesserung nur im "Bauchgefühl"

3 Die 3 Grading-Strategien

Wie bewertest du, ob ein Output "gut" ist? Es gibt drei Ansätze, jeder mit unterschiedlichem Trade-off zwischen Kosten, Geschwindigkeit und Flexibilität.

Strategie Wie es funktioniert Kosten Geschwindigkeit Flexibilität Wann nutzen
Rules-Based String-Match, Regex, JSON-Schema Minimal Sehr schnell Gering Strukturierte Outputs, Pflichtfelder
Claude-as-Judge Claude bewertet Output 1-5 Mittel Mittel Hoch Qualitative Bewertung, Tonalität
Hybrid Rules für Struktur + Claude für Qualität Mittel Mittel Hoch Empfohlen für die meisten Use-Cases

Strategie A: Rules-Based Grading

// Beispiel: Chatbot muss Preisinfo + Kontakt + CTA enthalten
function rulesBasedGrade(output: string): boolean {
  const checks = [
    output.includes('€') || output.includes('Euro'),  // Preis erwähnt
    output.match(/\d{2}:\d{2}/) !== null,               // Öffnungszeiten
    output.toLowerCase().includes('kontakt'),            // Kontakt-CTA
    output.length >= 100 && output.length <= 800,        // Länge OK
    !output.includes('ich weiß nicht'),                  // Kein Unwissen
  ]
  return checks.filter(Boolean).length >= 4 // mind. 4/5 müssen passen
}

// Regex-Beispiel: JSON-Format validieren
function validateJsonOutput(output: string): boolean {
  try {
    const parsed = JSON.parse(output)
    return (
      typeof parsed.title === 'string' &&
      Array.isArray(parsed.features) &&
      parsed.features.length >= 3
    )
  } catch {
    return false
  }
}

Strategie B: Claude-as-Judge

const judgePrompt = `Du bist ein Quality-Assurer für einen Kundensupport-Chatbot.

Bewerte folgende Antwort auf einer Skala von 1-5:
1 = Falsch/Gefährlich
2 = Unvollständig/Missverständlich
3 = Akzeptabel aber nicht optimal
4 = Gut, erfüllt alle Anforderungen
5 = Exzellent, übertrifft Erwartungen

Kriterien:
- Korrektheit der Information
- Freundlicher, professioneller Ton
- Klarer nächster Schritt für den Nutzer
- Keine Fehlinformationen

Kundenanfrage: "${evalCase.input}"
Bot-Antwort: "${output}"

Antworte NUR mit einer Zahl (1-5) und einer Begründung in einer Zeile.
Format: "SCORE: [1-5] | REASON: [kurze Begründung]"`

const judgment = await claude.messages.create({
  model: 'claude-haiku-4-5',  // Günstigeres Modell für Grading
  max_tokens: 100,
  messages: [{ role: 'user', content: judgePrompt }]
})

const scoreMatch = judgment.content[0].text.match(/SCORE:\s*(\d)/)
return parseInt(scoreMatch?.[1] ?? '0') >= 4

4 Das vollständige Eval-Skript (TypeScript)

Hier ist das komplette, produktionsbereite Eval-Skript das du für deine Projekte verwenden kannst:

import Anthropic from '@anthropic-ai/sdk'

const client = new Anthropic()

// ---- Typen ----
interface EvalCase {
  id: string
  input: string
  expectedOutput?: string
  checkFn?: (output: string) => boolean
  category: string
  priority: 'high' | 'medium' | 'low'
}

interface EvalResult {
  id: string
  passed: boolean
  output: string
  category: string
  priority: string
}

// ---- Eval-Fälle laden ----
const evalCases: EvalCase[] = [
  {
    id: 'e001',
    input: 'Wie viel kostet das Basic-Paket?',
    checkFn: (out) => out.includes('800') && out.includes('€'),
    category: 'pricing',
    priority: 'high'
  },
  {
    id: 'e002',
    input: 'Wann sind Sie erreichbar?',
    checkFn: (out) => out.match(/\d{1,2}[:\-]\d{2}/) !== null,
    category: 'support',
    priority: 'high'
  },
  {
    id: 'e003',
    input: 'Wie lange dauert das Onboarding?',
    expectedOutput: 'Woche',
    category: 'onboarding',
    priority: 'medium'
  },
  // ... weitere Cases
]

// ---- Eval-Runner ----
async function runEvals(
  cases: EvalCase[],
  systemPrompt: string
): Promise<{ score: number; total: number; passed: number; results: EvalResult[]; breakdown: Record }> {

  const results: EvalResult[] = []
  const categoryScores: Record = {}

  for (const evalCase of cases) {
    const response = await client.messages.create({
      model: 'claude-haiku-4-5',
      max_tokens: 512,
      system: systemPrompt,
      messages: [{ role: 'user', content: evalCase.input }]
    })

    const output = response.content[0].type === 'text'
      ? response.content[0].text
      : ''

    const passed = evalCase.checkFn
      ? evalCase.checkFn(output)
      : output.toLowerCase().includes(evalCase.expectedOutput!.toLowerCase())

    results.push({ id: evalCase.id, passed, output, category: evalCase.category, priority: evalCase.priority })

    // Category-Tracking
    if (!categoryScores[evalCase.category]) {
      categoryScores[evalCase.category] = { passed: 0, total: 0 }
    }
    categoryScores[evalCase.category].total++
    if (passed) categoryScores[evalCase.category].passed++
  }

  const passed = results.filter(r => r.passed).length
  const breakdown = Object.fromEntries(
    Object.entries(categoryScores).map(([cat, data]) => [
      cat, Math.round((data.passed / data.total) * 100)
    ])
  )

  return { score: passed / cases.length, total: cases.length, passed, results, breakdown }
}

// ---- Haupt-Ausführung ----
async function main() {
  const systemPromptV1 = `Du bist ein freundlicher Kundenberater für MeineAgentur GmbH...`
  const systemPromptV2 = `Du bist ein präziser Kundenberater für MeineAgentur GmbH.
  Antworte immer mit konkreten Zahlen und einem klaren nächsten Schritt...`

  console.log('🔬 Eval-Loop startet...\n')

  const baseline = await runEvals(evalCases, systemPromptV1)
  console.log(`📊 Baseline (v1): ${(baseline.score * 100).toFixed(1)}% (${baseline.passed}/${baseline.total})`)
  console.log('   Breakdown:', baseline.breakdown)

  const newVersion = await runEvals(evalCases, systemPromptV2)
  console.log(`\n📊 Neue Version (v2): ${(newVersion.score * 100).toFixed(1)}% (${newVersion.passed}/${newVersion.total})`)
  console.log('   Breakdown:', newVersion.breakdown)

  const delta = ((newVersion.score - baseline.score) * 100).toFixed(1)
  const improved = newVersion.score > baseline.score

  console.log(`\n${improved ? '✅' : '❌'} Delta: ${improved ? '+' : ''}${delta}%`)
  console.log(improved ? '→ Commit neue Version' : '→ Revert — Baseline war besser')

  // Fehlgeschlagene Cases ausgeben
  const failed = newVersion.results.filter(r => !r.passed)
  if (failed.length > 0) {
    console.log(`\n❌ Fehlgeschlagene Cases (${failed.length}):`)
    failed.slice(0, 5).forEach(f => {
      console.log(`  [${f.id}] ${f.category}/${f.priority}: "${f.output.slice(0, 80)}..."`)
    })
  }
}

main().catch(console.error)

5 Continuous Eval — GitHub Actions

Jeder Commit der den System-Prompt ändert löst automatisch Evals aus. Regressionen werden im PR sichtbar, bevor sie in Produktion gehen.

# .github/workflows/eval.yml
name: AI Eval Pipeline

on:
  pull_request:
    paths:
      - 'src/prompts/**'
      - 'src/ai/**'
  push:
    branches: [main]

jobs:
  eval:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      - name: Setup Node
        uses: actions/setup-node@v4
        with:
          node-version: '20'

      - name: Install dependencies
        run: npm ci

      - name: Run Eval Suite
        env:
          ANTHROPIC_API_KEY: ${{ secrets.ANTHROPIC_API_KEY }}
        run: npx tsx src/evals/run.ts --output json > eval-results.json

      - name: Check Eval Score
        run: |
          SCORE=$(cat eval-results.json | jq '.score')
          THRESHOLD=0.80
          echo "Eval score: $SCORE (threshold: $THRESHOLD)"
          if (( $(echo "$SCORE < $THRESHOLD" | bc -l) )); then
            echo "❌ Eval score below threshold! Blocking merge."
            exit 1
          fi
          echo "✅ Eval score OK"

      - name: Comment PR with results
        if: github.event_name == 'pull_request'
        uses: actions/github-script@v7
        with:
          script: |
            const results = require('./eval-results.json')
            github.rest.issues.createComment({
              issue_number: context.issue.number,
              owner: context.repo.owner,
              repo: context.repo.repo,
              body: `## 🔬 Eval Results\n\n**Score: ${(results.score * 100).toFixed(1)}%**\n\nBreakdown: ${JSON.stringify(results.breakdown)}`
            })
Ergebnis: Jeder PR zeigt automatisch den Eval-Score. Ein Rückgang über 3% blockiert den Merge. Das ist CI/CD für AI — so wie du es für Code kennst.

6 Eval-Patterns für verschiedene Use-Cases

Use-Case Grading-Strategie Key-Checks Min-Cases
FAQ-Chatbot Hybrid Korrekte Infos, kein Halluzinieren, CTA vorhanden 50
Klassifikator Rules-Based Exakter Label-Match, Precision/Recall 100
Text-Generator Claude-as-Judge Tonalität, Vollständigkeit, Keine Verbote 20
Code-Generator Rules-Based Kompiliert, Tests bestehen, Kein eval() 30
Data-Extraktor Rules-Based JSON-Schema-Valid, Alle Felder befüllt 40
Email-Generator Hybrid Betreff vorhanden, Tonalität, Keine Tippfehler 20

Eval-Set wachsen lassen: Jeder Bug = neuer Eval-Case

Die goldene Regel: Wenn ein Nutzer über einen schlechten Output beschwert, oder du einen Fehler findest — füge sofort einen neuen Eval-Case hinzu bevor du den Bug fixst. So vermeidest du Regressions-Blindheit. 
# Workflow bei Bug
1. Bug gefunden: Bot antwortet falsch auf "Habt ihr ein Studenten-Rabatt?"
2. Neuen Eval-Case hinzufügen (BEVOR Fix):
   { input: "Habt ihr einen Studentenrabatt?", checkFn: out => out.includes('nein') }
3. Eval laufen: zeigt Fehler (baseline sinkt)
4. Prompt fixen
5. Eval laufen: Case jetzt grün
6. Commit

7 Kunden-Eval-Sets aufbauen

Deine Kunden sind die besten Quellen für Eval-Cases — sie wissen, was "gut" bedeutet für ihr Business.

Der Kunden-Eval-Prozess

  1. Liefere Chatbot-Prototype an Kunden
  2. Bitte Kunden um 20 "gute Antworten" — Beispiele die er als perfekt empfinden würde. Format: Frage + Erwartete Antwort
  3. Automatisiere daraus Eval-Cases — Rules-Based wo möglich, Claude-as-Judge für Qualität
  4. Zeige dem Kunden den Score — "Euer Chatbot trifft 89% aller Erwartungen"
  5. Optimiere auf Score, nicht auf Gefühl — jede Iteration zeigt messbaren Fortschritt
Bonus: Der Eval-Score ist ein hervorragendes Verkaufsargument für deine Betreuung. "Ich habe euren Chatbot von 68% auf 91% Qualitäts-Score verbessert" ist überzeugender als "Ich habe den Prompt ein bisschen angepasst".

8 Anti-Patterns beim Eval-Aufbau

Anti-PatternProblemFix
Zu kleines Eval-Set (<10) Nicht repräsentativ, Ergebnis zufällig Minimum 20, ideal 50+
Nur manuelle Evals Nicht skalierbar, nicht in CI/CD integrierbar Automatisiertes Skript von Anfang an
Eval-Set nie updaten Regressionen auf neuen Use-Cases unbemerkt Jeder Bug → neuer Eval-Case
Teuerstes Modell für Grading Hohe Eval-Kosten, verlangsamt Loop Haiku für Grading, Sonnet für Production
Kein Score-Tracking Kein Trend sichtbar, Fortschritt unklar Scores in CSV oder DB loggen
Überfitting auf Eval-Set Score hoch, Production-Qualität trotzdem schlecht Eval-Set regelmäßig mit neuen Real-Cases erweitern

🎯 Praxis-Übung: Eval-Loop für deinen Chatbot

  1. Schritt 1: Wähle einen bestehenden Chatbot/Prompt-basierten Service den du für einen Kunden betreibst (oder baue einen Demo-Chatbot).
  2. Schritt 2: Erstelle ein Eval-Set mit mindestens 20 Cases. Mix: 10 Happy-Path, 5 Edge Cases, 5 Adversarial. Speichere als evals/cases.ts.
  3. Schritt 3: Implementiere das Eval-Skript aus dieser Lektion. Führe Baseline-Messung durch.
  4. Schritt 4: Ändere den System-Prompt in einer Richtung (z.B. mehr Beispiele, klarere Instruktionen). Führe Evals erneut aus. Vergleiche Scores.
  5. Schritt 5: Iteriere mindestens 3-4 Mal bis du einen Score-Gewinn von mindestens 10 Prozentpunkten erzielst.
  6. Bonus: Integriere die Evals als GitHub Action. Stelle sicher, dass PRs mit Prompt-Änderungen den Score automatisch anzeigen.
Erfolgskriterium: Du kannst dem Kunden sagen: "Wir haben euren Chatbot durch 4 Iterationen von X% auf Y% Qualitäts-Score verbessert." Zahlen statt Gefühl.
