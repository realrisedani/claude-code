Daten-Leaks

Lernziele

  • Eine Multi-Client-Struktur mit isoliertem Ordner und eigener .claude/-Config je Kunde aufsetzen
  • Eine klare Naming-Konvention etablieren, die Verwechslungen ausschließt
  • Verstehen, warum Kundendaten und Secrets niemals zwischen Kunden vermischt werden dürfen
Konzept — Ein Ordner pro Kunde, harte Grenzen

Ab dem dritten Kunden wird Ordnung zur Überlebensfrage. Wenn Kunde A und Kunde B im selben Verzeichnis liegen, ist es eine Frage der Zeit, bis ein Secret, ein Dateiname oder eine Antwort beim falschen Kunden landet. Das ist kein "Versehen" — das ist ein Strukturfehler.

Die Regel ist simpel: Ein Ordner pro Kunde, mit eigener .claude/-Config. Jeder Kunde ist eine geschlossene Welt — eigene Permissions, eigene Secrets, eigene CLAUDE.md, eigene MCP-Server. Die einzige Brücke nach außen ist die geteilte Knowledge Base (aus L41), die bewusst nur unkritische Vorlagen enthält.

Pro Kunde isoliert

  • Eigener Ordner
  • Eigene .claude/
  • Eigene Secrets
  • Eigene MCP-Server

Geteilt (read-only)

  • Agentur-KB
  • Templates, SOPs
  • Keine Kundendaten
  • Nur generische Vorlagen

Beim Wechsel

  • /clear zwischen Kunden
  • Anderen Ordner öffnen
  • Kein Context-Bleeding
  • Frischer Start
💡 Merksatz: Die Trennung passiert auf Datei-Ebene, nicht im Kopf. Wenn Kunde-B-Daten physisch nicht im Kunde-A-Ordner liegen, kann Claude sie auch nicht versehentlich heranziehen. Struktur schlägt Disziplin.
Die Ordnerstruktur

Ein Workspace, darin ein Ordner je Kunde, getrennt von der geteilten KB. Jeder Kundenordner hat seine eigene .claude/.

agentur-workspace/
├── agentur-kb/ # geteilt, read-only — nur generische Vorlagen (L41)
│
├── clients/
│ ├── baeckerei-sonne/ # Kunde A — geschlossene Welt
│ │ ├── .claude/
│ │ │ ├── settings.json # eigene Permissions
│ │ │ └── settings.local.json # Secrets — NIE committen
│ │ ├── CLAUDE.md # Kunde-A-Kontext
│ │ ├── .env # API-Keys von Kunde A
│ │ └── src/
│ │
│ ├── kanzlei-meier/ # Kunde B — komplett getrennt
│ │ ├── .claude/
│ │ ├── CLAUDE.md
│ │ └── .env # API-Keys von Kunde B
│ │
│ └── shop-nordlicht/ # Kunde C
│ └── ...
│
└── .gitignore # **/.env, **/settings.local.json
🚫 Die eine Regel, die alles zusammenhält: Kunde-A-Secrets dürfen NIEMALS in den Kunde-B-Kontext gelangen. Kein gemeinsames .env, kein geteiltes Notion-Token, kein "ich kopier das schnell rüber". Jeder Leak ist ein Vertrauensbruch — und je nach Branche ein Rechtsproblem (DSGVO).

💡 Naming-Konvention: Kundenordner immer kleinbuchstaben-mit-bindestrich, nie Kürzel oder Nummern. baeckerei-sonne ist eindeutig — kunde-3 oder BS führt früher oder später zur Verwechslung.
Per-Kunde settings.json

Jeder Kunde bekommt eigene Permissions. So kann Claude bei Kunde A z.B. deployen, bei Kunde B aber nur lesen.

# clients/baeckerei-sonne/.claude/settings.json
{
  "permissions": {
    "allow": ["Read", "Edit(src/**)", "Bash(npm run *)"],
    "deny": ["Read(../**)", "Edit(.env*)"]
  }
}
💡 Der deny: ["Read(../**)"]-Trick: Damit verbietest du Claude, aus dem Kundenordner nach oben in die Geschwister-Ordner zu lesen. Selbst wenn du dich vertippst — Claude kommt technisch nicht an die Daten von Kunde B.
Szenario — Drei Kunden parallel, keine Verwechslung

Was schiefgeht, wenn alles in einem Ordner liegt

1
Du arbeitest an Kunde A, Claude hat dessen .env mit Notion-Token im Kontext.
2
Ohne /clear wechselst du das Thema zu Kunde B — der alte Kontext bleibt hängen.
3
Claude generiert für Kunde B eine Mail und zieht aus dem Kontext versehentlich Kunde-A-Daten.
4
Du schickst die Mail raus — Kunde B sieht interne Infos von Kunde A. Vertrauen weg, evtl. DSGVO-Fall.
Der saubere Ablauf
/clear # Kontext leeren
cd ../kanzlei-meier && claude # neuen Kundenordner öffnen
# Claude startet frisch — nur Kunde-B-Kontext, nichts von A
✅ Goldene Regel: Ein Kundenwechsel ist immer ein /clear + Ordnerwechsel. Nie das Thema mitten in einer Session wechseln. Frischer Ordner = frischer Kontext = kein Leak.
Praxis — Kunden-Ordnerstruktur aufsetzen
# 1. Workspace + clients-Ordner
mkdir -p agentur-workspace/clients && cd agentur-workspace

# 2. Globale .gitignore — Secrets ausschließen
echo "**/.env" >> .gitignore
echo "**/.claude/settings.local.json" >> .gitignore

# 3. Ersten Kunden anlegen (Naming-Konvention!)
mkdir -p clients/baeckerei-sonne/.claude
touch clients/baeckerei-sonne/CLAUDE.md
touch clients/baeckerei-sonne/.claude/settings.json

# 4. Im Kundenordner starten
cd clients/baeckerei-sonne && claude
