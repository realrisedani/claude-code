Lernziele

  • Die 4 Pflicht-Artefakte einer sauberen Übergabe kennen: README, Runbook, Zugänge-Liste, Wartungs-Anleitung
  • Eine Übergabe-Checkliste abarbeiten, mit der ein Außenstehender in unter einem Tag produktiv wird
  • Ein wiederverwendbares Handoff-Doku-Template selbst erstellen
Konzept — Übergabe ist kein Abschiedsgeschenk

Du lieferst ein Projekt ab — und drei Wochen später kommt die erste Support-Mail: "Wie ändere ich den Text im Chatbot?" Dann die nächste: "Der Bot antwortet nicht mehr, was tun?" Ohne saubere Übergabe wirst du zum dauerhaften Hotline-Dienst für ein Projekt, für das du längst bezahlt wurdest.

Eine gute Übergabe macht den Kunden selbstständig. Sie beantwortet die Fragen, bevor sie gestellt werden. Das Ziel: Der Kunde (oder ein fremder Entwickler, den er beauftragt) versteht das Projekt in unter einem Tag — ohne dich anzurufen.

💡 Merksatz: Wissen im Kopf ist kein geliefertes Projekt. Erst wenn alles schriftlich und getestet ist, ist die Arbeit wirklich übergeben. Alles andere ist eine unsichtbare Verbindlichkeit, die dich später Zeit kostet.
Die 4 Pflicht-Artefakte
ArtefaktZweckBeantwortet die Frage
README.mdEinstieg + lokaler Start"Was ist das und wie bringe ich es zum Laufen?"
RUNBOOK.mdBetrieb + Fehlerbehebung"Es geht etwas nicht — was tue ich?"
Zugänge-ListeAccounts, Keys, Rollen"Wo logge ich mich ein und mit welchen Rechten?"
Wartungs-AnleitungInhalte/Prompts pflegen"Wie ändere ich Texte, ohne etwas kaputt zu machen?"
💡 Zugänge-Liste — Sonderregel: Sie enthält keine Passwörter im Klartext, sondern verweist auf einen Passwort-Manager (z.B. 1Password-Vault) oder einen sicheren Übergabe-Kanal. Niemals Keys in eine committe README schreiben.
README-Struktur
# Bäckerei Sonne — FAQ-Chatbot

## Was ist das?
Ein FAQ-Chatbot auf der Website, der Kundenfragen zu
Öffnungszeiten, Sortiment und Bestellungen beantwortet.

## Schnellstart
npm install
cp .env.example .env # Keys eintragen (siehe Zugänge-Liste)
npm run dev # läuft auf http://localhost:3000
npm run deploy # live stellen

## Voraussetzungen
- Node 20+, Anthropic-API-Key
- Hosting: Vercel (Account in Zugänge-Liste)

## Inhalte ändern
Texte des Bots liegen in content/faq.md — siehe WARTUNG.md
Runbook-Struktur
# RUNBOOK — Betrieb & Notfälle

## Was tun, wenn der Bot nicht antwortet?
1. Status-Seite des Hosters prüfen (Link)
2. npm run logs — Fehlermeldung lesen
3. API-Key abgelaufen? → in Vercel erneuern (siehe Zugänge)

## Was tun, wenn eine Antwort falsch ist?
1. Frage in content/faq.md suchen und korrigieren
2. npm run deploy — Änderung geht live

## Routine: monatlicher Check
- Logs auf Fehler durchsehen
- 5 Test-Fragen stellen, Antworten prüfen

## Notfall-Kontakt
Agentur: name@agentur.de — Reaktionszeit 24h (Wartungsvertrag)
✅ Die eine Regel: Nichts wird übergeben, ohne dass du die Anleitung selbst einmal befolgt hast. Jeder Befehl in README und Runbook muss von dir auf einer frischen Umgebung getestet sein. Eine ungetestete Anleitung ist schlimmer als keine — sie erzeugt falsches Vertrauen.
Szenario — Sauberer Handoff, zufriedener Kunde

Projektende bei Bäckerei Sonne — zwei Wege

1
Du übergibst Repo + README + Runbook + Zugänge-Liste + Wartungs-Anleitung, alle getestet.
2
Eine Woche später will der Kunde einen FAQ-Eintrag ändern — er schlägt in WARTUNG.md nach und schafft es allein.
3
Der Bot hakt einmal — der Kunde folgt dem Runbook, erneuert den Key, alles läuft. Kein Anruf bei dir.
4
Der Kunde empfiehlt dich weiter: "Die haben so sauber übergeben, da musste ich nie nachfragen."
Der Gegenentwurf
Ohne Doku: 6 Support-Mails im ersten Monat, jede kostet dich 20 Minuten unbezahlt. Saubere Übergabe ist nicht nur netter — sie schützt deine Zeit und macht aus Kunden Empfehler.
🚫 Die häufigste Falle: "Ich erkläre das schnell im Call." Mündliche Übergaben verpuffen. Der Kunde vergisst 80% nach drei Tagen. Ein Handoff-Call ergänzt die Doku — er ersetzt sie nie.
Praxis — Handoff-Doku-Template erstellen

Erstelle ein wiederverwendbares Template in deiner KB (aus L41), das du bei jedem Projektende nur noch ausfüllst.

# 1. Template in der KB anlegen
mkdir -p agentur-kb/templates/handoff
cd agentur-kb/templates/handoff
touch README.md RUNBOOK.md WARTUNG.md ZUGAENGE.md

# 2. Mit Claude die Gerüste füllen
claude # "Erstelle README-, Runbook- und Wartungs-Gerüste mit Platzhaltern"

# 3. Bei jedem Projektende kopieren + ausfüllen
cp -r agentur-kb/templates/handoff/* clients/baeckerei-sonne/docs/

# 4. Anleitungen auf frischer Umgebung TESTEN, dann committen
git add docs/ && git commit -m "handoff: doku fuer baeckerei sonne"
