Konzept — Warum eine zentrale KB

Du arbeitest mit fünf Kunden. Bei jedem schreibst du dieselbe Onboarding-Mail, baust dieselbe Projektstruktur, erklärst dasselbe Pricing. Jedes Mal von vorn. Das ist der Solo-Macher-Engpass: dein Wissen steckt in deinem Kopf und in verstreuten Ordnern. Niemand sonst — auch kein Claude — kann darauf zugreifen.

Die Knowledge Base dreht das um. Ein einziges Repo enthält alles, was bei jedem Kunden gleich ist: deine Prozesse, deine bewährten Prompts, deine Vorlagen, dein Pricing, deine Standard-Antworten. Claude liest dieses Repo als Kontext und liefert dir in Minuten eine angepasste Vorlage statt eines leeren Blatts.

Ohne KB

  • Wissen im Kopf
  • Jeder Kunde von null
  • Nicht delegierbar
  • Inkonsistente Qualität

Verstreute Docs

  • Notion hier, Drive da
  • Claude findet nichts
  • Veraltete Versionen
  • Copy-Paste-Fehler

Zentrale KB

  • Ein versioniertes Repo
  • Claude zieht es als Kontext
  • Eine Wahrheit, alle aktuell
  • Sofort delegierbar
💡 Merksatz: Die KB ist nicht "noch ein Ordner". Sie ist das Betriebssystem deiner Agentur. Alles, was du zweimal machst, gehört einmal in die KB — und danach nie wieder neu erfunden.

Wie Claude auf die KB zugreift

Das Herzstück ist die CLAUDE.md im KB-Root. Sie ist die Landkarte: Claude liest sie zuerst und weiß dann, wo welches Wissen liegt.

# Agentur Knowledge Base — Einstiegspunkt

Dieses Repo ist die zentrale Wissensbasis der Agentur.
Nutze es als Vorlagen-Quelle für JEDEN Kundenauftrag.

## Wo liegt was
- sops/ → Abläufe (Onboarding, Kickoff, Handoff)
- prompts/ → bewährte System-Prompts
- templates/ → Mail-, Angebots-, README-Vorlagen
- pricing/ → Pakete und Konditionen
- faq/ → Standard-Antworten + Einwand-Behandlung

## Regel
Bei neuem Auftrag IMMER zuerst hier nach einer Vorlage
suchen, dann auf den Kunden anpassen. Nie bei null starten.
💡 Zwei Wege, die KB heranzuziehen: Entweder du arbeitest direkt im KB-Repo, oder du referenzierst es aus dem Kundenprojekt — z.B. mit @../agentur-kb/templates/angebot.md. Bei der API packst du die relevanten KB-Dateien als cached System-Prompt davor (90% günstiger bei vielen Anfragen).

Szenario — Neuer Auftrag, Vorlage in Minuten

Montag, 9:00 — Neukunde "Bäckerei Sonne" will einen FAQ-Chatbot

1
Du legst clients/baeckerei-sonne/ an und startest Claude im Ordner.
2
Du sagst: "Nimm das Chatbot-Prompt-Template aus der KB und passe es auf eine Bäckerei an."
3
Claude liest agentur-kb/prompts/chatbot-system.md und liefert eine angepasste Version.
4
Du sagst: "Und das Standard-Angebot für Paket M." Claude zieht templates/angebot.md + pricing/pakete.md.
5
Nach 10 Minuten hast du Prompt + Angebot fertig — statt eines halben Tages bei null.
Der Hebel
Ohne KB ist jeder Neukunde ein halber Tag Setup. Mit KB sind es Minuten. Bei 5 Neukunden im Quartal sparst du Tage — und lieferst konsistente Qualität.
🚫 Don'ts gehören explizit in die KB: Schreib auf, was die Agentur nie tut — z.B. "Niemals konkrete ROI-Versprechen geben", "Niemals Wettbewerberpreise nennen", "Niemals ohne Vertrag starten". Claude hält sich an explizite Don'ts, rät aber bei Lücken.
Praxis — KB-Grundgerüst anlegen
# 1. KB-Repo anlegen
mkdir agentur-kb && cd agentur-kb
git init

# 2. Bereiche als Ordner
mkdir memory sops prompts templates pricing faq

# 3. Einstiegspunkt + erste Dateien
touch CLAUDE.md sops/onboarding.md templates/angebot.md pricing/pakete.md

# 4. Claude im Ordner starten und befüllen lassen
claude # "Schreibe sops/onboarding.md aus meinem aktuellen Ablauf"

# 5. Committen — die KB ist versioniert
git add . && git commit -m "kb: grundgeruest mit 6 bereichen"
✅ Goldene Regel: Erst dokumentieren, dann skalieren. Eine KB, die in deinem Kopf bleibt, kann nie ein Mitarbeiter oder ein Agent übernehmen. Schriftlich = übergebbar.
