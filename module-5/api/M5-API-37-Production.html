Lernziele

  • Die drei Säulen produktiver KI-Systeme unterscheiden: Reliability, Cost, Security
  • Einen Retry-Wrapper mit Exponential Backoff für Rate-Limits und Server-Fehler bauen
  • Prinzipien für Monitoring, Kostenkontrolle und sicheren Umgang mit Secrets übertragen
Konzept — Prototyp ist nicht Produktion
💡 Der Unterschied: Ein Prototyp macht einen Call und hofft, dass er klappt. Ein Produktionssystem rechnet damit, dass Calls fehlschlagen, Limits greifen und Kosten explodieren — und fängt das ab. Diese Lektion ist Mindset, keine neue API.

Reliability

  • Retries mit Backoff
  • Rate-Limits (429) abfangen
  • Timeouts + Idempotenz

Cost

  • usage jedes Calls loggen
  • günstigeres Modell wählen
  • Caching + max_tokens-Limit

Security

  • Key aus Env, nie im Code
  • User-Input nie blind vertrauen
  • Keys nie ins Frontend
Reliability — Retry mit Exponential Backoff
🚫 Falle: Bei einem 429 (Rate-Limit) oder 529 (überlastet) sofort und stur neu zu feuern macht es schlimmer. Du musst warten — und mit jedem Versuch länger (exponentiell).
import time, random
from anthropic import Anthropic, RateLimitError, APIStatusError
client = Anthropic()

def create_with_retry(**kwargs):
  for attempt in range(5):
    try:
      return client.messages.create(**kwargs)
    except (RateLimitError, APIStatusError) as e:
      if attempt == 4:
        raise # nach 5 Versuchen aufgeben
      wait = (2 ** attempt) + random.random() # 1s, 2s, 4s, 8s + Jitter
      print(f"Retry in {wait:.1f}s ({e.__class__.__name__})")
      time.sleep(wait)

resp = create_with_retry(
  model="claude-sonnet-4-5", max_tokens=512,
  messages=[{"role": "user", "content": "Hallo!"}],
)
✅ Drei Bausteine eines guten Retries: exponentiell wachsende Wartezeit, ein zufälliger Jitter (verhindert, dass alle Clients gleichzeitig wiederkommen), und ein Limit an Versuchen, damit du nicht ewig hängst.
Cost & Security — Prinzipien als Code
import os, logging

# Security: Key NUR aus der Env, niemals hardcoden
client = Anthropic(api_key=os.environ["ANTHROPIC_API_KEY"])

resp = create_with_retry(
  model="claude-sonnet-4-5",
  max_tokens=512, # Cost: harte Obergrenze pro Antwort
  messages=[{"role": "user", "content": "..."}],
)

# Cost: usage strukturiert loggen (Monitoring)
logging.info("claude_call", extra={
  "in": resp.usage.input_tokens,
  "out": resp.usage.output_tokens,
  "model": resp.model,
})
🚫 Tödlicher Fehler: Den API-Key ins Frontend (JS im Browser) packen oder ins Git committen. Jeder kann ihn auslesen und auf deine Rechnung Calls feuern. Keys gehören ausschließlich serverseitig in Env-Variablen.

Szenario — Vom Skript zum Service

Szenario: Dein Bot geht von 10 auf 10.000 Nutzer

1
Reliability: Jeder Call läuft durch create_with_retry — 429/529 werden abgefangen
2
Cost: usage jedes Calls wird geloggt, du siehst die Tagesausgaben in Echtzeit
3
Cost: häufiger Kontext liegt im Cache (L32), max_tokens deckelt jede Antwort
4
Security: Key kommt aus der Env, User-Input wird validiert, nichts läuft im Browser
Das Prinzip
Produktion heißt: mit Fehlern rechnen, Kosten messen, Geheimnisse schützen. Wer das von Anfang an einbaut, skaliert ohne böse Überraschungen.
💡 Transfer: Geh dein letztes API-Skript durch und frag für jede der drei Säulen: Was passiert bei einem 429? Wo sehe ich meine Kosten? Wo liegt mein Key? Jede Antwort, die "weiß nicht" lautet, ist deine nächste Aufgabe.
