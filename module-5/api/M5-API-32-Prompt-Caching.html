Konzept — Warum cachen
💡 Das Problem: Bei jedem Call schickst du oft denselben großen System-Prompt oder dieselbe Wissensbasis mit — und zahlst sie jedes Mal voll. Caching speichert diesen Teil serverseitig. Folgende Calls innerhalb der TTL lesen ihn fast gratis.
from anthropic import Anthropic
client = Anthropic()

BIG_CONTEXT = open("handbuch.txt").read() # z.B. 5000 Tokens

resp = client.messages.create(
  model="claude-sonnet-4-5", max_tokens=512,
  system=[
    {"type": "text", "text": "Du bist ein Support-Bot."},
    {
      "type": "text",
      "text": BIG_CONTEXT,
      "cache_control": {"type": "ephemeral"} # <- Breakpoint
    },
  ],
  messages=[{"role": "user", "content": "Wie storniere ich?"}],
)
🚫 Mindestgröße beachten: Caching greift erst ab einer Mindest-Token-Zahl pro Block. Sehr kurze System-Prompts lassen sich nicht cachen — dann gibt es keinen Hit und du zahlst normal. Cachen lohnt sich nur bei großen, wiederkehrenden Blöcken.
Cache-Hits messen
print(resp.usage.cache_creation_input_tokens) # beim 1. Call: Cache wird gefüllt
print(resp.usage.cache_read_input_tokens) # bei Folge-Calls: > 0 = Hit!
print(resp.usage.input_tokens) # nur die NICHT gecachten Tokens
✅ So liest du es: Erster Call → cache_creation_input_tokens ist hoch (Cache wird geschrieben). Zweiter Call innerhalb der TTL → cache_read_input_tokens ist hoch, input_tokens ist klein. Genau das ist die Ersparnis.

💡 Rechenbeispiel: 5000-Token-Kontext, 1000 Anfragen/Tag. Ohne Cache zahlst du 5000 Tokens × 1000 mal voll. Mit Cache zahlst du ihn ~1.25x einmal pro 5 Min und danach nur ~0.1x — das summiert sich schnell zu rund 90% Ersparnis auf dem gecachten Teil.
Wo Caching lohnt

Stark

  • großer System-Prompt
  • Wissensbasis / Doku
  • fixe Tool-Definitionen

Mittel

  • lange Gesprächshistorie
  • wiederkehrende Few-Shots
  • Code-Kontext einer Datei

Lohnt nicht

  • kurze, variable Prompts
  • einmalige Anfragen
  • unter Mindest-Token
Szenario — Support-Bot mit Handbuch

Szenario: 1000 Support-Anfragen am Tag, gleiches Handbuch

1
Du legst das 5000-Token-Handbuch als eigenen text-Block in system
2
Du setzt cache_control: {"type":"ephemeral"} auf diesen Block
3
1. Anfrage füllt den Cache — cache_creation_input_tokens ist hoch
4
Alle Anfragen in den nächsten ~5 Min lesen ihn billig — cache_read_input_tokens > 0
Reihenfolge zählt
Gecachte Blöcke gehören an den ANFANG (System / fixer Kontext). Variable Teile (die User-Frage) kommen danach — sonst bricht der Cache.
💡 Jetzt du: Bau einen System-Prompt aus einem langen Text (Doku oder Handbuch), setz cache_control darauf. Schick zweimal hintereinander dieselbe Frage und vergleich cache_creation_input_tokens (Call 1) mit cache_read_input_tokens (Call 2).
