Konzept — Was ist MCP?
💡 Die Analogie: Vor USB brauchte jedes Gerät seinen eigenen Stecker. USB hat alles vereinheitlicht — ein Standard, jedes Gerät passt. MCP ist genau das für KI: ein Standard-Stecker, mit dem Claude an jedes externe Tool andocken kann — GitHub, dein Dateisystem, eine Datenbank, einen Browser.

Ohne MCP kann Claude nur Text lesen und schreiben. Mit MCP bekommt Claude Augen (er liest direkt aus deinen Tools) und Hände (er führt dort Aktionen aus). Ein MCP-Server ist ein kleines Programm, das eine Brücke zwischen Claude und einem Tool baut.

Was MCP konkret freischaltet

  • GitHub: Issues lesen, PRs kommentieren, CI-Status prüfen — ohne Browser
  • Filesystem: Dateien außerhalb des Projekt-Ordners lesen und schreiben
  • Datenbank: SQL-Queries ausführen und Ergebnisse direkt verarbeiten
  • Browser: Webseiten öffnen, klicken, Screenshots für UI-Tests machen
stdio vs. SSE — die zwei Server-Typen

stdio (lokal)

  • Läuft als lokaler Prozess auf deinem Rechner
  • Start über command + args
  • Schnell, kein Netzwerk nötig
  • Standard für die meisten Server

SSE / HTTP (remote)

  • Läuft auf einem entfernten Server
  • Verbindung über URL
  • Für Cloud-Dienste + Teams
  • Braucht oft ein Auth-Token

Wann was?

  • Lokales Tool → stdio
  • Cloud-API → SSE
  • Im Zweifel: mit stdio starten
  • SSE später bei Bedarf
Praxis — Einen MCP-Server eintragen

In .claude/settings.json kommt ein Block mcpServers. Jeder Server bekommt einen Namen, ein command, args und optional env für Tokens:

{
  "mcpServers": {
    "filesystem": {
      "command": "npx",
      "args": [
        "-y",
        "@modelcontextprotocol/server-filesystem",
        "/Users/dani/projekte"
      ]
    }
  }
}

Ein Server mit Token (z. B. GitHub) bekommt zusätzlich einen env-Block:

{
  "mcpServers": {
    "github": {
      "command": "npx",
      "args": ["-y", "@modelcontextprotocol/server-github"],
      "env": {
        "GITHUB_PERSONAL_ACCESS_TOKEN": "ghp_xxx"
      }
    }
  }
}
🚫 Token-Falle: Server mit echten Tokens gehören in .claude/settings.local.json — niemals in die committed settings.json. Sonst landet dein Token öffentlich auf GitHub.
Server prüfen
# alle eingetragenen MCP-Server anzeigen
claude mcp list

# Details + Verbindungsstatus eines Servers
claude mcp get filesystem
✅ Regel: Nach jedem neuen Server-Eintrag claude mcp list ausführen. Steht der Server auf connected, ist er einsatzbereit. Steht er auf failed, stimmt etwas mit command, args oder Token nicht.
Warum sich der Aufwand lohnt

Szenario: Ein Bug-Report aus einem GitHub-Issue umsetzen

1
Ohne MCP: Du öffnest den Browser, suchst das Issue, kopierst den Text manuell in Claude
2
Claude schlägt einen Fix vor — du kopierst den Code zurück, testest, fixt, wiederholst
3
Am Ende kopierst du den Kommentar wieder per Hand ins Issue zurück
4
5 Tool-Wechsel, 4 Copy-Paste-Schritte, jedes Mal Fehlerquelle
Mit GitHub-MCP — alles in einem Befehl
Du tippst Lies Issue #42 und schlag einen Fix vor. Claude liest das Issue direkt, schreibt den Fix, kommentiert das Ergebnis ins Issue — ohne dass du den Browser öffnest. Kein Copy-Paste, keine Fehlerquelle.
💡 Jetzt du: Trage den filesystem-Server in deine .claude/settings.json ein (mit einem echten Pfad außerhalb deines Projekts). Führe dann claude mcp list aus und prüfe, ob er auf connected steht. Bitte Claude danach, eine Datei aus diesem externen Pfad zu lesen.
