Vor dem Commit

Lernziele

  • Den /code-review Skill vor jedem Commit als Reflex nutzen
  • Verstehen, was ein guter Review findet — Logik-Bugs, Edge-Cases, Secrets
  • Einen Bug vor dem Merge fangen statt in der Produktion
Konzept — Was ein Review prüft

Logik

  • Tut der Code, was er soll?
  • Off-by-one, falsche Bedingung
  • Vergessene Fälle
  • Race Conditions

Edge-Cases

  • Leere Liste, null, undefined
  • Sehr große Eingaben
  • Netzwerk-Fehler abgefangen?
  • Was bei 0 oder negativ?

Sicherheit

  • Secrets im Code?
  • SQL-Injection, XSS
  • Unvalidierte Eingaben
  • Zu offene Berechtigungen
💡 Review prüft nur den Diff: /code-review schaut sich genau deine geänderten Zeilen an, nicht das ganze Projekt. Das macht den Review schnell, günstig und fokussiert.
Praxis — Review vor dem Commit
# 1. Änderungen stagen
git add src/checkout.js

# 2. Review auf den gestagten Diff laufen lassen
/code-review

# oder mit höherem Effort für wichtige Änderungen
/code-review --effort high
# Beispiel-Findings, die ein Review zurückgibt:
[BUG] checkout.js:42 — bei leerem Warenkorb crasht total()
[SECURITY] checkout.js:18 — API-Key steht im Klartext im Code
[EDGE] checkout.js:55 — negativer Rabatt nicht abgefangen
[STYLE] checkout.js:30 — Funktion zu lang, aufteilen empfohlen
Review-Kategorien im Überblick
KategorieFindetBeispiel
Logik-BugFalsches VerhaltenSumme ohne Steuer berechnet
Edge-CaseVergessene SonderfälleLeere Liste crasht die App
SicherheitSecrets, InjectionPasswort im Code committed
StilLesbarkeit, DuplikationGleiche Logik dreimal kopiert
⚠ Review ersetzt keine Tests: Ein Review liest den Code, führt ihn aber nicht aus. Er fängt viele Bugs — aber nur echte Tests beweisen, dass es wirklich läuft. Beides zusammen nutzen.
Bug vor dem Merge gefangen

Szenario: Review rettet die Produktion

1
Du baust einen Rabatt-Code in den Checkout. Sieht gut aus, du willst committen
2
Vorher: git add und dann /code-review
3
Der Review meldet: bei einem Rabatt über 100% wird der Preis negativ — kein Check vorhanden
4
Du fügst den Check hinzu, reviewst nochmal, dann committest du. Kein Geld verschenkt in Produktion
Die Regel — Reflex vor jedem Commit
git add <dateien>
/code-review # Findings durchgehen + fixen
git commit -m "..."
✅ Reflex aufbauen: /code-review wird so selbstverständlich wie git status. Ein günstiger medium-Review vor jedem Commit kostet Sekunden und fängt teure Fehler.
