Konzept — Ein Ort für alles
💡 Kerngedanke: Alles aus den letzten Lektionen — Permissions (L3), MCP-Server (L19/20), Hooks (L22) — landet in einer Datei: settings.json. Wer diese Datei beherrscht, konfiguriert Claude einmal sauber und arbeitet danach reibungslos. Das ist der Profi-Hebel.
Die 3 Ebenen — wer gewinnt?

Global (User)

  • ~/.claude/settings.json
  • Gilt in allen Projekten
  • Deine Grund-Defaults
  • Schwächste Ebene

Projekt (Team)

  • .claude/settings.json
  • Gilt im Repo, fürs Team
  • Wird committet
  • Überschreibt global

Lokal (privat)

  • .claude/settings.local.json
  • Nur du, deine Secrets
  • Nie committen
  • Gewinnt immer
✅ Merksatz: Je näher am Projekt und an dir, desto stärker. Reihenfolge: global < projekt < lokal. Eine Einstellung in settings.local.json überschreibt dieselbe Einstellung aus den anderen beiden.

Praxis — eine vollständige settings.json

So sieht eine durchdachte Projekt-.claude/settings.json aus — alle sechs Keys an einem Ort:

{
  "model": "claude-sonnet-4-5",
  "env": { "NODE_ENV": "development" },
  "permissions": {
    "allow": [ "Read", "Edit(src/**)", "Bash(npm run *)" ],
    "deny":  [ "Edit(.env*)", "Bash(rm -rf*)" ],
    "ask":   [ "Bash(git push:*)" ]
  },
  "hooks": {
    "PostToolUse": [
      { "matcher": "Write|Edit",
        "hooks": [ { "type": "command", "command": "npx prettier --write . || true" } ] }
    ]
  },
  "mcpServers": {
    "filesystem": { "command": "npx",
      "args": [ "-y", "@modelcontextprotocol/server-filesystem", "/Users/dani/projekte" ] }
  },
  "enabledPlugins": [ "realrise-tools@1.0.0" ]
}

Secrets (Tokens, persönliche Overrides) gehören dagegen in die lokale Datei:

# .claude/settings.local.json — NUR du, nie ins Repo
{
  "mcpServers": {
    "github": { "command": "npx",
      "args": [ "-y", "@modelcontextprotocol/server-github" ],
      "env": { "GITHUB_PERSONAL_ACCESS_TOKEN": "ghp_xxx" } }
  }
}
🚫 Eiserne Regel: .claude/settings.local.json wird niemals committet. Sie enthält deine Tokens und persönlichen Overrides. Schütze sie sofort per .gitignore, sonst landen deine Secrets öffentlich im Repo.
echo ".claude/settings.local.json" >> .gitignore
git add .gitignore .claude/settings.json
git commit -m "claude: projekt-settings + local ignoriert"
Reihenfolge in Aktion

Szenario: Eine Einstellung an drei Stellen — welche zählt?

1
Global steht model: claude-opus — deine Grundvorgabe für alle Projekte
2
Im Projekt steht model: claude-sonnet-4-5 — das Team hat sich auf Sonnet geeinigt
3
In deiner local-Datei steht model: claude-haiku — du willst hier billig testen
4
Es gilt Haiku — die lokale Ebene gewinnt immer, egal was global oder projekt sagt
Die Lehre
Wenn eine Einstellung „nicht greift", ist fast immer eine stärkere Ebene schuld. Prüf zuerst settings.local.json — sie überschreibt alles.
💡 Jetzt du: Setze eine vollständige .claude/settings.json für ein Projekt auf — mit model, permissions (allow + deny) und einem PostToolUse-Hook. Lege Tokens in settings.local.json, schütze sie per .gitignore und committe nur die geteilte Datei.
