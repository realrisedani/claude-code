Lernziele

  • Verstehen, warum parallele Tool-Calls schneller und günstiger sind als sequenzielle

Konzept — Was Parallelität bedeutet

Wenn Claude drei Dateien lesen muss, kann er sie entweder nacheinander öffnen (lesen, warten, lesen, warten, lesen) oder alle drei in einem Schritt anfordern. Das Zweite ist parallel: ein einziger Turn, alle Ergebnisse kommen zusammen zurück. Voraussetzung — die Aktionen hängen nicht voneinander ab.

💡 Tipp: Stell dir einen Koch vor. Drei Töpfe gleichzeitig auf den Herd ist parallel. Erst Topf 1 fertig kochen, dann Topf 2 anstellen ist sequenziell — und dauert dreimal so lang, obwohl die Töpfe nichts miteinander zu tun haben.
Parallel vs. Sequenziell

Parallel OK

  • Mehrere Reads verschiedener Files
  • Mehrere Greps nach Mustern
  • Edits auf verschiedene Files
  • Mehrere Sub-Agents starten

Faustregel

  • Braucht B das Ergebnis von A?
  • Nein → parallel
  • Ja → sequenziell
  • Gleiche Datei schreiben → nacheinander

Sequenziell nötig

  • Edit dann Read derselben Datei
  • git add dann git commit
  • Befehl braucht Output des vorigen
  • Schreiben auf gleiche Datei
🚫 Falle: Abhängige Befehle parallel anstoßen führt zu Fehlern. git commit ohne vorheriges git add committet nichts — die Reihenfolge ist Pflicht.
So löst du Parallelität aus

Du brauchst keinen technischen Befehl. Ein klar formulierter Prompt mit dem Wort "gleichzeitig" oder "parallel" reicht.

# Prompt der Parallelität auslöst:
Lies gleichzeitig package.json, tsconfig.json und
.eslintrc und fass mir die Setup-Konfiguration zusammen.
# Noch ein Beispiel — parallele Suche:
Durchsuch das Projekt parallel nach allen Stellen, die
"fetchUser" verwenden, und allen, die "AuthToken" verwenden.
✅ Regel: Bei Discovery-Aufgaben (verstehen, suchen, mehrere Dateien lesen) immer "gleichzeitig" oder "parallel" in den Prompt — Claude bündelt die Calls in einen Turn.
Szenario — der Unterschied in der Praxis

Codebase verstehen: sequenziell vs. parallel

1
Du fragst: "Versteh die Auth" — Claude liest Datei 1, wartet, liest Datei 2, wartet, liest Datei 3
2
Fünf Dateien, fünf Turns — jede Wartezeit summiert sich, der Token-Overhead pro Turn auch
3
Anders: "Lies alle fünf Auth-Dateien gleichzeitig und erklär mir das System"
4
Ein Turn, alle fünf Dateien kommen zusammen zurück — spürbar schneller, weniger Kosten
Merke
Parallel lohnt sich genau dann, wenn die Aktionen nichts voneinander wissen müssen. Das ist bei Lesen und Suchen fast immer der Fall.
