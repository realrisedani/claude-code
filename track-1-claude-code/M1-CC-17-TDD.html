Lernziele

Konzept — Red, Green, Refactor

TDD dreht die Reihenfolge um: erst der Test, dann der Code. Du beschreibst, was rauskommen soll, Claude schreibt den Test (er schlägt fehl, weil es noch keine Implementierung gibt), und dann schreibt er den minimalen Code, bis der Test grün ist. Zum Schluss räumt er auf, während die Tests grün bleiben.

💡 Tipp: Der Test ist die Spezifikation in ausführbarer Form. Wenn Claude weiß, was der Test verlangt, hat er keinen Interpretationsspielraum mehr — er muss genau das bauen, was der Test prüft.
Warum TDD mit Claude so stark ist

Test = klare Spec

  • Kein Raten, was gemeint ist
  • Claude liebt klare Vorgaben
  • Erwartung ist messbar

Halluzination fällt auf

  • Erfundener Code → Test rot
  • Du siehst es sofort
  • Kein stilles Falschverhalten

Du tippst nichts

  • Claude schreibt Test und Code
  • Du diktierst nur die Fälle
  • Du prüfst nur grün/rot
Praxis — der TDD-Zyklus

Zuerst lässt du Claude nur den Test schreiben — ausdrücklich ohne Implementierung. Beispiel: eine Funktion slugify, die Text in URL-Slugs verwandelt.

# Red: Test zuerst, ohne Implementierung
Schreib NUR einen Test für eine Funktion slugify(text).
Fälle: "Hallo Welt" → "hallo-welt", Umlaute "Über" → "ueber",
leerer String → "". Keine Implementierung. Lass den Test laufen.
# Der Test (von Claude) — schlägt erwartbar fehl, slugify fehlt noch
test('macht aus Text einen Slug', () => {
  expect(slugify('Hallo Welt')).toBe('hallo-welt');
  expect(slugify('Über')).toBe('ueber');
  expect(slugify('')).toBe('');
});

Test ist rot. Jetzt darf Claude den minimalen Code schreiben — bis grün.

# Green: minimale Implementierung bis der Test grün ist
function slugify(text) {
  return text
    .replace(/ü/g, 'ue')
    .toLowerCase()
    .trim()
    .replace(/\s+/g, '-');
}
✅ Regel: Erst den Test rot sehen, dann erst Code. Lass dir das rote Ergebnis zeigen — sonst weißt du nicht, ob der Test überhaupt etwas prüft.
🚫 Falle: Wenn Claude Test und Code in einem Rutsch liefert und beides direkt grün ist, hast du nie gesehen, ob der Test wirklich fehlschlägt. Verlange "zeig mir den Test rot" als eigenen Schritt.
Szenario — TDD deckt eine Lücke auf

Der vergessene Edge-Case

1
Du diktierst die Fälle, vergisst aber den Slug mit mehreren Leerzeichen
2
Claude baut slugify, alle deine Tests sind grün
3
Du ergänzt den Test: "Hallo   Welt" soll "hallo-welt" geben — Test wird rot
4
Claude bessert den Regex nach, Test grün — die Lücke ist geschlossen, bevor sie in Produktion ging
Der Wert von TDD
Jeder neue Fall, den du als Test formulierst, wird sofort geprüft. Du baust deine Spec Stück für Stück aus — und Claude muss jeder Erweiterung folgen.
Jetzt du — selbst ausprobieren
💡 Jetzt du: Such dir ein kleines Feature (z. B. eine Validierungs- oder Formatierungsfunktion). Lass Claude zuerst nur den Test mit drei bis vier Fällen schreiben und rot laufen. Dann erst den minimalen Code bis grün. Ergänze danach einen Edge-Case-Test und beobachte, wie Claude den Code nachzieht. Du tippst dabei keine Code-Zeile selbst.
