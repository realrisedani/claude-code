Praxis — Mehrere Server in einer settings.json

Alle Server stehen im selben mcpServers-Block. Tokens kommen in env — und dieser ganze Block gehört in .claude/settings.local.json, nicht ins Repo:

{
  "mcpServers": {
    "github": {
      "command": "npx",
      "args": ["-y", "@modelcontextprotocol/server-github"],
      "env": { "GITHUB_PERSONAL_ACCESS_TOKEN": "ghp_xxx" }
    },
    "filesystem": {
      "command": "npx",
      "args": ["-y", "@modelcontextprotocol/server-filesystem", "/Users/dani/projekte"]
    },
    "brave-search": {
      "command": "npx",
      "args": ["-y", "@modelcontextprotocol/server-brave-search"],
      "env": { "BRAVE_API_KEY": "BSA_xxx" }
    },
    "apify": {
      "command": "npx",
      "args": ["-y", "@apify/actors-mcp-server"],
      "env": { "APIFY_TOKEN": "apify_api_xxx" }
    }
  }
}
🚫 Tokens nie ins Repo: Dieser komplette Block enthält 3 echte Tokens. Liegt er in der committed settings.json, sieht ihn jeder, der das Repo klont — und kann auf deine GitHub-, Brave- und Apify-Konten zugreifen. Tokens gehören ausschließlich in .claude/settings.local.json.
# local-settings dauerhaft aus git ausschließen
echo ".claude/settings.local.json" >> .gitignore
git add .gitignore && git commit -m "gitignore: claude local settings"

# prüfen, dass die Server stehen
claude mcp list
✅ Regel: Server-Struktur (welche Server, welche args) darf ins Repo, damit das Team sie kennt. Server-Secrets (env-Tokens) niemals. Trenne beides sauber.
Use-Case — Recherche-zu-Code in einem Flow

Szenario: Wettbewerber-Reels analysieren und ins Repo dokumentieren

1
Du bittest Claude, mit Apify die letzten 20 Reels eines Konkurrenz-Profils zu scrapen
2
Claude nutzt Web-Search, um Trends rund um die Top-Hooks zu prüfen
3
Claude schreibt die Analyse als Markdown ins Projekt — über den Filesystem-Server
4
Per GitHub-Server öffnet Claude direkt ein Issue mit den nächsten Content-Ideen
Das Ergebnis
Vier Tools, ein durchgehender Flow, kein einziger Browser-Wechsel. Genau das macht die Server zu Must-Haves für eine Agentur.
💡 Jetzt du: Hol dir ein GitHub Personal Access Token (Settings → Developer settings → Tokens). Trage den github-Server mit dem Token in .claude/settings.local.json ein. Prüfe mit claude mcp list, dass er connected ist, und bitte Claude, deine offenen Issues aufzulisten.
