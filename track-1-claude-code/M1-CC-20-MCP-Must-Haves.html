Konzept — Warum genau diese 6?
💡 Ohne MCP kann Claude Code nur lokale Dateien lesen und Terminal-Befehle ausführen. Mit diesen 6 Servern deckt eine Agentur alles ab: Versionierung, Dateisystem, Projektmanagement, Wissen, Browser und persistentes Gedächtnis. Jeder Server öffnet eine neue Dimension.
Die 6 Server im Detail
GitHub PAT-Token
  • Issues + PRs direkt aus Claude öffnen & steuern
  • Code-Review über mehrere Repos ohne Browser
  • CI-Status abfragen, Branch-Vergleiche, Release-Notes

Vollständiges Konfig-Recipe (settings.json)

Alle 6 Server in einer Datei. Struktur darf committed werden — Tokens gehören in settings.local.json:

// ~/.claude/settings.json — Struktur (commitbar) { "mcpServers": { "github": { "type": "stdio", "command": "npx", "args": ["-y", "@modelcontextprotocol/server-github"], "env": { "GITHUB_PERSONAL_ACCESS_TOKEN": "ghp_..." } }, "filesystem": { "type": "stdio", "command": "npx", "args": ["-y", "@modelcontextprotocol/server-filesystem", "/Users/dani/projekte"] }, "notion": { "type": "stdio", "command": "npx", "args": ["-y", "@notionhq/notion-mcp-server"], "env": { "NOTION_API_KEY": "secret_..." } }, "linear": { "type": "stdio", "command": "npx", "args": ["-y", "@linear/mcp"], "env": { "LINEAR_API_KEY": "lin_api_..." } }, "memory": { "type": "stdio", "command": "npx", "args": ["-y", "@modelcontextprotocol/server-memory"] }, "playwright": { "type": "stdio", "command": "npx", "args": ["-y", "@executeautomation/playwright-mcp-server"] } } }
🚫 Tokens niemals ins Repo: Alle env-Werte mit echten Tokens gehören in .claude/settings.local.json — diese Datei muss in .gitignore stehen. Wer das Repo klont, sieht die Struktur, aber keine Secrets.
Server verbinden & prüfen
# settings.local.json absichern echo ".claude/settings.local.json" >> .gitignore # Alle verbundenen Server anzeigen claude mcp list # Einzelnen Server testen (Beispiel GitHub) claude "Zeig mir meine letzten 5 offenen GitHub Issues"
Permissions-Disziplin

Minimal-sufficient bedeutet: jeder Server bekommt nur die Rechte, die er wirklich braucht. Kein Server soll schreiben dürfen, der nur lesen muss.

Read-only Server

  • GitHub (Analyse)
  • Memory (nur lesen)
  • Linear (Issues einsehen)

Read + Write

  • Filesystem (bewusst!)
  • Notion (Pages anlegen)
  • Memory (speichern)

Gefährlich — prüfen

  • GitHub (PRs mergen)
  • Filesystem (/)
  • Browser (Formulare)
⚠ Filesystem-Pfad einschränken: Gib nie / oder /Users als Root an — immer den spezifischsten Pfad, z. B. /Users/dani/projekte. Sonst kann Claude theoretisch jede Datei auf dem System lesen oder verändern.

CLAUDE.md Integration

Dokumentiere deine MCPs in der CLAUDE.md — jedes Teammitglied sieht sofort, welche Server verfügbar sind und was sie dürfen:

## MCP Server (global verfügbar) | Server | Typ | Zugriff | Token in | |------------|--------|--------------------------------|-----------------------| | github | stdio | Repos, Issues, PRs | settings.local.json | | filesystem | stdio | /Users/dani/projekte (rw) | kein Token | | notion | stdio | Team-Workspace (rw) | settings.local.json | | linear | stdio | alle Projekte (r) | settings.local.json | | memory | stdio | persistent Knowledge-Graph | kein Token | | playwright | stdio | Headless Chrome (Vorsicht!) | kein Token | ## Wichtige Regeln - Tokens NIEMALS in settings.json (nur in settings.local.json) - Filesystem: nur /Users/dani/projekte — kein Slash-Root - Playwright nur für explizit genehmigte Aufgaben nutzen
✅ Warum das wichtig ist: In 3 Monaten weißt du nicht mehr, welcher Server welche Rechte hat. Das Team noch weniger. Eine saubere CLAUDE.md-Dokumentation ist die einzige verlässliche Quelle — nicht das Gedächtnis.
Live-Szenario — Agentur-Workflow mit 3 Servern

Szenario: Kunden-Brief zu Linear-Ticket zu GitHub-PR

1
Claude liest via Notion-MCP den aktuellen Kunden-Brief aus dem Workspace
2
Auf Basis des Briefs öffnet Claude via Linear-MCP direkt die Tickets mit Aufwandsschätzung
3
Claude erstellt via GitHub-MCP den Feature-Branch und die erste Commit-Struktur
4
Alle Entscheidungen werden via Memory-MCP persistent für spätere Sessions gespeichert
Das Ergebnis
Vom Kunden-Brief zum laufenden Branch in einem einzigen Claude-Prompt — ohne Browser-Wechsel. Das ist der Unterschied zwischen einem netten Chat-Tool und einer echten Agentur-Infrastruktur.
Deine Live-Übung heute
🛠 Übung 1 — GitHub live: Hol dir ein Personal Access Token (GitHub → Settings → Developer settings → Tokens → fine-grained). Trag den GitHub-Server in settings.local.json ein. Starte Claude neu, prüfe mit claude mcp list und bitte Claude: "Zeig mir meine letzten 3 Repos mit offenen Issues."
🛠 Übung 2 — Memory live: Trag den Memory-Server ein (kein Token nötig). Sage Claude in einem Prompt: "Speichere, dass mein Kunden-Standardpreis 1.500 € pro Monat ist." Schließe Claude, öffne neu, frag: "Was ist mein Kunden-Standardpreis?" — er soll es noch wissen.
