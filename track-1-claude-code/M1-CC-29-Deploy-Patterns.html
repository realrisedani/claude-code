Lernziele

  • Sicher deployen — mit Plan, nicht aus dem Bauch
  • Eine Rollback-Strategie haben, die in unter 5 Minuten zurückrollt
  • Nach dem Deploy beobachten und aus Incidents im Postmortem lernen
Konzept — Die 3 Phasen jedes Deploys

Deploy

  • Erst Staging, dann Live
  • Tests grün, Rollback-Plan da
  • Klein und oft statt selten gross
  • Danach sofort beobachten

Rollback

  • Plan steht vor dem Deploy
  • Zurück in unter 5 Minuten
  • Vorherige Version promoten
  • Oder git revert + redeploy

Postmortem

  • Was ist passiert?
  • Ohne Schuldzuweisung
  • Welche Massnahme verhindert es?
  • Im Repo dokumentieren
💡 Rollback schlägt Hotfix: Wenn die Produktion brennt, ist der schnellste Weg zurück fast immer die alte Version wieder live nehmen — nicht hektisch einen Fix bauen. Erst stabil, dann in Ruhe die Ursache finden.
Praxis — sicher deployen und zurückrollen
# Sicher ausrollen: erst Tests, dann Deploy
npm test && npm run build # nur bei grün weiter
git tag v1.4.0 && git push --tags # Version markieren
npx vercel --prod # live gehen
# Rollback Variante A — vorherige Version promoten (schnellster Weg)
vercel rollback # Vercel: letztes Deploy zurück
netlify rollback # Netlify: vorheriges Deploy live

# Rollback Variante B — über git, wenn kein Promote-Button da ist
git revert HEAD --no-edit # fehlerhaften Commit rückgängig
git push # Pipeline deployt die heile Version

✅ Goldene Regel: Nie Freitagabend ohne Rollback-Plan deployen. Geht etwas schief, willst du am Wochenende zurückrollen können — nicht stundenlang debuggen, während keiner erreichbar ist.
Deploy geht schief — schneller Rollback

Szenario: Live-Bug nach dem Deploy

1
Du deployst v1.4.0. Zwei Minuten später melden Kunden: der Checkout lädt nicht mehr
2
Du analysierst nicht stundenlang — du rollst zurück: vercel rollback bringt v1.3.9 wieder live
3
Unter 5 Minuten nach dem Bug ist die alte, stabile Version wieder online. Kunden können wieder kaufen
4
In Ruhe: Ursache finden, Fix bauen, testen. Danach Postmortem schreiben, damit es nicht wieder passiert
Die Regel — Rollback zuerst, Analyse danach
vercel rollback # 1. stabil machen
# 2. Ursache in Ruhe finden
# 3. Postmortem: was, warum, welche Massnahme
🚫 Nicht im Live-System debuggen: Solange Kunden betroffen sind, gilt: erst zurückrollen, dann analysieren. Jede Minute Ausfall kostet Vertrauen — und manchmal Geld.
