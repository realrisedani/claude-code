Lernziele

  • Atomare Commits schreiben — ein logischer Change pro Commit
  • Gute Commit-Messages formulieren, die das Warum erklären
  • Mit Feature-Branches arbeiten und git status vor riskanten Aktionen prüfen
Konzept — Was einen Profi-Commit ausmacht

Atomar

  • Ein logischer Change pro Commit
  • Bug-Fix und Feature getrennt
  • Leicht zu reviewen
  • Sauber zurückrollbar

Aussagekräftig

  • Subject erklärt warum
  • Max 72 Zeichen, kein Punkt am Ende
  • Body bei Bedarf für Kontext
  • Deutsch oder Englisch — konsistent

Isoliert

  • Arbeit läuft im Feature-Branch
  • main bleibt immer sauber
  • PR bündelt zusammenhängende Commits
  • Nach Merge: Branch löschen
💡 Claude als Commit-Partner: Sag "committe das atomar mit einer Message, die erklärt warum". Claude trennt unzusammenhängende Änderungen automatisch in mehrere Commits — wenn du es darum bittest.
Gute vs. schlechte Commit-Messages
Schlecht (was)Gut (warum)Warum besser
update codefix(login): leite bei abgelaufener session zum login umSagt was kaputt war
changesfeat(export): csv-export für rechnungen ergänztNennt das Feature
fix bugfix(api): null-antwort vom server abfangenBeschreibt den Bug
wiprefactor(user): validierung in eigene funktion ausgelagertErklärt den Umbau
✅ Goldene Regel: Ein logischer Change pro Commit. Wenn du in der Message ein "und" brauchst ("login fix und neues icon"), sind das zwei Commits.
Praxis — sauberer Feature-Branch-Flow
# 1. Erst prüfen: ist der Stand sauber?
git status # sollte "nothing to commit" zeigen

# 2. Feature-Branch von main abzweigen
git switch -c feature/csv-export

# 3. Nur die zusammengehörenden Dateien stagen
git add src/export.js src/export.test.js

# 4. Atomar committen mit gutem Subject
git commit -m "feat(export): csv-export für rechnungen ergänzt"
# 5. Branch pushen und Upstream setzen
git push -u origin feature/csv-export

# 6. PR direkt aus dem Terminal öffnen
gh pr create --title "CSV-Export für Rechnungen" \
  --body "Fügt CSV-Export hinzu. Warum: Kunde braucht Daten für Steuerberater."
⚠ Vor riskanten Aktionen immer git status: Vor git reset, git rebase oder Branch-Wechsel. Nur ein sauberer Stand schützt vor verlorenen Änderungen.
Chaotischer vs. sauberer Branch-Verlauf

Szenario: Zwei Wege durch dasselbe Feature

1
Chaos: Alles auf main. 1 Riesen-Commit "stuff" mit 30 Dateien — Feature, Bugfix und Tippfehler gemischt
2
Ein Reviewer versteht nichts. Der Bug lässt sich nicht isoliert zurückrollen, ohne das Feature mitzunehmen
3
Sauber: Branch feature/csv-export. 3 atomare Commits — Feature, Tests, Doc getrennt
4
PR ist in 5 Minuten reviewt. Jeder Commit einzeln verständlich, jeder Teil einzeln rückrollbar
Die Regel — vor jedem Feature
git switch -c feature/name # nie direkt auf main arbeiten
git add <nur zusammengehörende dateien>
git commit -m "<typ>(<scope>): <warum>"
🚫 Nie direkt auf main pushen, nie force-pushen: Auf geteilten Branches zerstört git push --force die Arbeit anderer. Force nur auf deinem eigenen Feature-Branch — und auch da nur bewusst.
