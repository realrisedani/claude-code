Lernziele

  • Refactoring immer mit grünen Tests als Sicherheitsnetz durchführen
  • In kleinen, atomaren Commits umbauen — ein Refactor pro Commit
  • Refactoring und Feature-Arbeit strikt trennen
Konzept — Was sauberes Refactoring ausmacht

Refactoring heißt: das Verhalten bleibt gleich, nur die Struktur wird besser. Der einzige Beweis, dass sich das Verhalten nicht geändert hat, sind grüne Tests. Deshalb gilt: vorher müssen die Tests grün sein, und nach jedem Schritt müssen sie grün bleiben. Wird ein Test rot, hast du das Verhalten kaputt gemacht — und du weißt sofort, an welchem kleinen Schritt es lag.

💡 Tipp: Kleine Schritte sind kein Komfort, sondern Sicherheit. Wer 200 Zeilen am Stück umbaut und am Ende einen roten Test sieht, sucht ewig. Wer in zehn kleinen Commits umbaut, weiß genau, welcher Commit den Test gekippt hat.
Die drei Disziplinen

Tests grün halten

  • Vorher grün, nachher grün
  • Nach jedem Schritt prüfen
  • Rot → sofort stoppen

Kleine Schritte

  • Ein Pattern pro Schritt
  • Rename, dann Extract, dann Split
  • Nie alles gleichzeitig

Atomare Commits

  • Ein Refactor = ein Commit
  • Commit-Message beschreibt einen Schritt
  • Leicht rückgängig zu machen
✅ Regel: Niemals Refactoring und neues Feature im selben Commit. Refactor-Commits ändern Struktur, Feature-Commits ändern Verhalten — vermischt sind beide unmöglich sauber zu reviewen oder zurückzudrehen.
Praxis — atomare Commits beim Umbau

Vor dem Start: Tests laufen lassen, grün bestätigen. Dann ein Pattern umsetzen, Tests erneut laufen, committen — und das pro Schritt wiederholen.

# Vor dem Refactor: Ausgangslage absichern
git status # sauber?
npm test # alle grün?
# Schritt 1: nur umbenennen, dann committen
# (Claude: "Benenne getUsr in getUser um, überall")
npm test # noch grün?
git add -A
git commit -m "refactor: getUsr in getUser umbenannt"

# Schritt 2: Funktion extrahieren, dann committen
npm test
git add -A
git commit -m "refactor: validierung in eigene funktion extrahiert"
🚫 Falle: Beim Umbenennen nicht nur den Code prüfen — auch Strings, Kommentare, Logs, Docs und Config-Dateien können den alten Namen enthalten. Lass Claude projektweit suchen, nicht nur die offensichtliche Datei.

Szenario — großer Umbau in kleinen Schritten

Eine 400-Zeilen-Datei entwirren

1
Tests laufen, alle grün — die Ausgangslage ist abgesichert
2
Schritt 1: unklare Variablennamen umbenennen → Tests grün → committen
3
Schritt 2: doppelten Block in eine Funktion extrahieren → Tests grün → committen
4
Schritt 3: Datei in zwei Module splitten → Tests grün → committen
Das Ergebnis
Drei saubere, atomare Commits statt eines riesigen unreviewbaren Klotzes. Wäre Schritt 2 schiefgegangen, hättest du nur diesen einen Commit verworfen — die anderen bleiben.
Jetzt du — selbst ausprobieren
💡 Jetzt du: Such dir eine unübersichtliche Funktion in deinem Projekt. Stell sicher, dass Tests dafür existieren und grün sind. Lass Claude in drei getrennten, atomaren Schritten refactoren (z. B. Rename, Extract, Split) — nach jedem Schritt Tests laufen lassen und einzeln committen. Mach in keinem dieser Commits ein neues Feature.
