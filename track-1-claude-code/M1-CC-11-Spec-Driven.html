Konzept — Warum Spec-Driven

Wenn du Claude nur "bau mir ein Login" sagst, muss er die Lücken füllen — und er rät. Mal richtig, oft daneben. Eine Spec ist eine kurze Beschreibung dessen, was du wirklich willst, BEVOR auch nur eine Zeile Code entsteht. Sie ist kein Bürokratie-Dokument, sondern dein günstigster Hebel gegen Re-Work. Bei der YC (Y Combinator) ist Spec-Driven Development Empfehlung Nummer eins für die Arbeit mit KI-Codern.

💡 Kerngedanke: Eine Spec besteht aus drei Fragen — was soll entstehen, warum brauchst du es, wie soll es grob funktionieren. Mehr nicht. Drei Sätze schlagen drei Stunden Nacharbeit.
Der Ablauf in 4 Schritten

1 · Spec

  • was / warum / wie
  • Kurz, in Worten
  • Noch kein Code

2 · Plan Mode

  • /plan aktivieren
  • Claude zeigt Vorgehen
  • Liest, schreibt nicht

3 · Bestätigen

  • Plan prüfen
  • Korrigieren falls nötig
  • Erst dann freigeben

Eine echte Mini-Spec

So sieht eine brauchbare Spec aus — als spec.md oder direkt im Prompt:

# Spec: E-Mail-Sammler für die Landing-Page

## Was
Ein Formular unten auf der Seite, das eine E-Mail entgegennimmt.

## Warum
Newsletter-Leads sammeln, bevor das Produkt fertig ist.

## Wie
- Ein Feld + Button "Eintragen"
- E-Mail an bestehenden Netlify-Forms-Endpoint
- Nach Absenden: Danke-Text statt Formular
- Design: bestehende CSS-Tokens, kein neues Framework

## Nicht jetzt
Double-Opt-In, Mailchimp-Anbindung — kommt später.
✅ Regel: Die Spec sagt das WAS und WARUM klar — beim WIE bleibst du grob. Du gibst die Richtung vor, nicht jede einzelne Code-Zeile. Sonst engst du Claude unnötig ein.
Ohne Spec vs. mit Spec

Szenario: Ohne Spec baut Claude am Ziel vorbei

1
Du tippst nur: "Bau ein Kontaktformular"
2
Claude rät: nimmt ein fremdes Framework, baut 5 Felder, eigenes Design
3
Du wolltest: 1 Feld, bestehender Endpoint, eure CSS-Tokens
4
Ergebnis: alles nochmal — 3 Stunden Nacharbeit verbrannt
Mit Spec — derselbe Task, direkt richtig
/plan # nach der Spec: Vorgehen zeigen lassen
# Plan prüfen → "passt, implementier exakt so"
# → Claude baut genau das Richtige, kein Re-Work
🚫 Falle — Spec überspringen "weil es schnell gehen muss": Genau dann rät Claude am meisten. Die 3 Sätze Spec sind nie der Zeitfresser — das falsch Gebaute ist es.
Praxis — Mini-Spec schreiben
💡 Jetzt du: Nimm ein kleines Feature, das du wirklich brauchst (z. B. ein Kontaktformular). Schreib in drei Blöcken was / warum / wie. Dann: /plan, Plan prüfen, freigeben, implementieren lassen. Vergleiche das Ergebnis mit deinem ersten Bauchgefühl.
