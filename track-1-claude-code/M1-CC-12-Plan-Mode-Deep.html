Lernziele

  • Plan Mode mit Shift+Tab aktivieren und seinen Read-Only-Charakter verstehen
  • Erkennen, bei welchen Aufgaben sich ein Plan lohnt und bei welchen er Zeitverschwendung ist
  • Einen Plan reviewen, freigeben und für das Team in .claude/plans/ committen
Konzept — Was Plan Mode ist

Im Plan Mode darf Claude nur lesen: Dateien öffnen, Code durchsuchen, die Struktur verstehen. Er schreibt nichts und führt keine Befehle aus, die etwas ändern. Am Ende bekommst du einen Plan — eine geordnete Liste, was er tun WÜRDE. Du liest sie, gibst frei, und erst dann fängt er an.

💡 Tipp: Plan Mode ist wie ein Architekt, der erst die Baupläne zeigt, bevor die Bagger anrollen. Du fängst keinen Schaden ein, weil noch nichts gebaut wurde.
Wann Plan Mode — wann nicht

Immer planen

  • Neue Architektur entwerfen
  • Unbekannte Codebase erkunden
  • Riskante Migration (DB, Auth)
  • Refactor über mehr als 5 Files

Abwägen

  • Mittlere Features, 2-3 Dateien
  • Du kennst den Code, aber nicht den Weg
  • Mehrere mögliche Ansätze
  • Kurz planen, dann los

Kein Plan nötig

  • Tippfehler, 1-Zeilen-Fix
  • Klares kleines Feature
  • Du weißt genau was kommt
  • Plan würde nur bremsen
🚫 Falle: Für jeden Mini-Task einen Plan anfordern bremst dich aus. Plan Mode ist für Größe und Risiko da — nicht für jeden Handgriff.
Praxis — Plan Mode aktivieren und nutzen

In der laufenden Session Shift+Tab drücken. Dann gibst du den Auftrag — Claude liest und plant, schreibt aber nichts.

# In der Session: Shift+Tab drücken → Plan Mode an
# Dann diesen Auftrag eintippen:

Analysiere den Ordner src/auth und erstelle einen Plan,
um die Session-Logik von Cookies auf JWT umzustellen.
Liste alle betroffenen Dateien und die Reihenfolge der Schritte.

Claude liefert den Plan. Wenn er gut ist, gibst du frei — und wechselst zurück in den Execute-Modus.

# Plan ist gut → freigeben
Der Plan passt. Setz Schritt 1 und 2 um, dann halt an
und zeig mir das Ergebnis bevor du weitermachst.

# Plan unvollständig → nachschärfen statt sofort starten
Im Plan fehlt die Rollback-Strategie und die Tests.
Ergänze beides, dann gib mir den Plan nochmal.
Pläne committen für Team-Review

Gute Pläne legst du als Markdown-Datei ins Repo. So kann das Team den Plan reviewen, bevor irgendwer Code schreibt — genau wie bei einem Pull Request.

# Plan-Ordner anlegen und Plan speichern
mkdir -p .claude/plans
# Claude bitten, den Plan als Datei abzulegen:
# "Schreib den Plan nach .claude/plans/jwt-migration.md"

git add .claude/plans/jwt-migration.md
git commit -m "plan: jwt-migration zur review"
✅ Regel: Bei riskanten Umbauten zuerst den Plan committen und reviewen lassen — dann erst umsetzen. Ein Plan im git ist nachvollziehbar; ein Bauchgefühl nicht.

Szenario — mit vs. ohne Plan

Großes Feature ohne Plan = Chaos

1
Du sagst "Bau die Zahlungsabwicklung um" ohne Plan
2
Claude fängt sofort an und ändert 9 Dateien quer durchs Projekt
3
Auf halbem Weg merkt ihr: die Datenbank-Migration fehlt komplett
4
Halb umgebaut, halb alt — nichts läuft, und du verstehst den Stand nicht mehr
Mit Plan läuft es sauber
Plan Mode hätte die fehlende Migration im Review aufgedeckt. Du hättest sie ergänzt, freigegeben, und Claude wäre Schritt für Schritt sauber durchgelaufen — mit Stopp-Punkten zum Prüfen.
Jetzt du — selbst ausprobieren
💡 Jetzt du: Such dir ein echtes Feature aus deinem Projekt, das mehrere Dateien betrifft. Drück Shift+Tab, lass Claude einen Plan erstellen, geh ihn mit dem 5-Punkte-Check durch. Erst wenn alle 5 Punkte stehen, gibst du frei. Committe den Plan nach .claude/plans/.

3 Fragen zum Nachdenken
