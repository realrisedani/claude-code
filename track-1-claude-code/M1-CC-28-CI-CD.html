Lernziele

  • GitHub Actions verstehen — Workflow, Jobs, Steps, Auslöser
  • Eine .github/workflows/ YAML-Datei lesen und anlegen
  • Secrets als GitHub Secrets verwalten — nie im YAML hardcoden
Konzept — Wie GitHub Actions aufgebaut ist

Trigger (on)

  • Wann läuft die Pipeline?
  • on: push, pull_request
  • Auch nach Zeitplan möglich
  • Oder manuell starten

Jobs

  • Eigenständige Arbeitspakete
  • Laufen auf einem Runner
  • z.B. ubuntu-latest
  • Können parallel laufen

Steps

  • Einzelne Befehle im Job
  • run: oder uses:
  • Checkout, Install, Test
  • Laufen nacheinander
💡 CI = Continuous Integration: Bei jedem Push prüft GitHub automatisch deinen Code — Tests, Lint, Build. CD = Continuous Deployment: Läuft alles grün, wird automatisch deployed. Claude kann beides als Action ausführen.
Praxis — eine Workflow-Datei

Datei anlegen: .github/workflows/ci.yml

name: CI
on:
  push:
    branches: [main]
  pull_request:

jobs:
  test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: "20"
      - run: npm ci
      - run: npm run lint
      - run: npm test
# Claude als Schritt im Workflow (Headless)
      - run: npm install -g @anthropic-ai/claude-code
      - run: claude -p "Schreibe die PR-Beschreibung aus dem Diff"
        env:
          ANTHROPIC_API_KEY: ${{ secrets.ANTHROPIC_API_KEY }}
⚠ Secrets niemals ins YAML: Der API-Key kommt aus ${{ secrets.ANTHROPIC_API_KEY }}. Den echten Wert legst du unter Repo → Settings → Secrets ab. Im YAML steht nur der Verweis — sonst landet dein Key öffentlich im Repo.
Die CI-Stages
StageBefehlPrüft
Lintnpm run lintStil und offensichtliche Fehler
Testnpm testFunktioniert die Logik?
Buildnpm run buildLässt sich das Projekt bauen?
Deploynpm run deployGeht es live (nur bei grün)?
✅ Reihenfolge zählt: Erst Lint, dann Test, dann Build, zuletzt Deploy. Bricht eine frühe Stage ab, läuft der Rest gar nicht — du sparst Zeit und deployst nie kaputten Code.
Push löst die Pipeline aus

Szenario: Vom Push zur grünen Pipeline

1
Du committest ein Feature und pushst auf deinen Branch
2
GitHub erkennt den Push, liest .github/workflows/ci.yml und startet den Job auf einem Runner
3
Steps laufen nacheinander: checkout → npm ci → lint → test. Alles ohne dein Zutun
4
Pipeline grün → PR ist merge-bereit. Rot → du siehst sofort welcher Step gebrochen ist
Die Regel — Workflow lokal anlegen
mkdir -p .github/workflows
# ci.yml anlegen (siehe oben)
git add .github/workflows/ci.yml && git commit -m "ci: pipeline aufgesetzt"
git push # Pipeline läuft beim ersten Push
🚫 Nie den echten API-Key committen: Auch nicht "nur kurz zum Testen". Ein gepushter Key ist sofort kompromittiert. Immer über GitHub Secrets.
Jetzt du
💡 Praxis — Minimal-Workflow anlegen: Bitte Claude: "Leg .github/workflows/ci.yml an, das bei jedem Push npm ci und npm test ausführt." Push danach und schau im Actions-Tab, ob die Pipeline grün läuft.
