5 Pflicht-Elemente

  • Problem-Statement — was ist das Problem?
  • User-Story — wer will was warum?
  • Acceptance Criteria — wann ist es fertig?
  • Tech-Constraints — was darf nicht geändert werden?
  • Out-of-Scope — was gehört explizit nicht dazu?

Spec committen

  • /specs/feature-name.md im Repo
  • Nachvollziehbar: warum was gebaut wurde
  • Claude liest Spec → keine Missverständnisse
  • Review-fähig wie Code

Mini-Spec für Bug-Fixes

  • Bug: was passiert falsch
  • Erwartet: was sollte passieren
  • Reproduce: Schritte zum Nachstellen
  • 3 Zeilen reichen — kein Overhead

Anti-Pattern Over-Spec

  • Nicht jede Code-Zeile spezifizieren
  • Architektur ja — Implementation nein
  • Zu detaillierte Spec = Claude eingeengt
  • Ziel: Klarheit, nicht Micromanagement
💡 Dialog-Beispiel: "Schreib eine Spec für X" → Review → "Implementiere Spec exakt" → kein Re-Work
🎯 Lernziel: Spec-Driven-Loop gefahren, committed Spec, Implementation passt 100%
